## Unifier Dimensions

Five dimensions not covered by the journals above.

### D1 â€” Widget Empty States

**Scope:** Widget front-face content â€” the "no data" / "add items to begin" state shown when a widget has no content to display.

**Canon:** Use `<ScaledEmptyState icon={IconName} title="..." subtitle="..." />` from `components/common/ScaledEmptyState.tsx`. Never hand-roll a centered div with an icon and text. The shared component handles container-query scaling, icon sizing, and consistent visual tone automatically.

**Reference usage:**

```tsx
import { ScaledEmptyState } from '@/components/common/ScaledEmptyState';
import { Clock } from 'lucide-react';

<ScaledEmptyState
  icon={Clock}
  title="No Schedule"
  subtitle="Flip to add schedule items."
/>;
```

**Key distinction:** Info-only panels (e.g., First5/Settings.tsx) that are always shown and not triggered by missing data are NOT empty states â€” leave them alone.

### D2 â€” Brand Color CSS Variables in Inline Styles

**Scope:** Any widget or component that sets brand blue/red as a background or foreground color via an inline `style={{}}` prop.

**Canon:** Use `var(--spart-primary, #2d3f89)` for brand blue and `var(--spart-accent, #ad2122)` for brand red. The fallback value preserves the existing color if the admin has not customized the theme. Hardcoding `#2d3f89` in an inline style bypasses the admin's global style configuration.

**Set by:** `DashboardView.tsx` reads admin config and writes `--spart-primary` / `--spart-accent` as CSS custom properties on the root dashboard container.

**Key distinction:** User-configurable color picker defaults (e.g., Countdown `eventColor = '#2d3f89'`, MaterialsWidget `titleColor = '#2d3f89'`) are correct as-is â€” they are user-owned values, not theme references. Color palette arrays (NextUp/Settings color picker, CollectionColorPicker) are swatch definitions, not theme references. CSS gradient stops in design components (BannerInteraction) are intentional. Only fix inline `backgroundColor` / `color` that should visually track the admin's theme.

**Status (run 4):** Dimension is ALIGNED. All `#ad2122` instances are intentional exceptions. All `#2d3f89` inline style instances are fixed or intentional exceptions. `MusicWidget/Widget.tsx:348` fallback color is D2-E5; `Countdown/Widget.test.tsx` assertion is a test file, not production code. No further D2 work needed until a new widget introduces the anti-pattern.

### D3 â€” Settings Panel Label Primitives

**Scope:** Labels above form controls in widget Settings components (back-face flip panel) AND admin configuration panels. Both use the same label primitive pattern.

**Canon:** Use `<SettingsLabel>Label Text</SettingsLabel>` from `components/common/SettingsLabel.tsx`. This encapsulates `text-xxs font-black text-slate-400 uppercase tracking-widest block mb-2` into a single component. Do not hand-roll this class combination.

**Optional icon:** `<SettingsLabel icon={Clock}>Label</SettingsLabel>` for labeled sections with icons.
**HTML for:** `<SettingsLabel htmlFor="my-input">Label</SettingsLabel>` when labeling a form input.

**Key distinction:** Descriptive body text and help text (`text-xs text-slate-400` explanatory paragraphs) are NOT SettingsLabel â€” leave them as `<p>` tags. Only the small uppercase section/field labels should use SettingsLabel. Button text (`font-black uppercase tracking-widest` on `<button>` elements) is NOT SettingsLabel â€” leave it alone. Colored labels inside themed info/callout boxes (emerald, brand-blue) are intentionally distinct â€” leave them alone.

### D4 â€” Import Path Convention (Cross-Directory)

**Scope:** Import statements that reference modules in a different directory from the importing file.

**Canon:** Use the `@/` alias for all cross-directory imports. The alias maps to the repo root. Examples:

- `import { WidgetLayout } from '@/components/widgets/WidgetLayout'` âś“
- `import { Toggle } from '@/components/common/Toggle'` âś“
- `import { useDashboard } from '@/context/useDashboard'` âś“

Relative `'../foo'` imports are acceptable ONLY within a flat sibling context (same directory, e.g., `import { X } from './helper'`). Multi-level `'../foo'` reaching into a different feature directory is the anti-pattern.

**Key distinction:** True same-directory imports (`./`) and direct parent sibling (`'../WidgetLayout'` from inside a widget subfolder) sit in a gray zone. The dominant pattern for WidgetLayout is `@/components/widgets/WidgetLayout` â€” prefer that form on new code and when touching a file anyway.

**Status (run 11):** All known cross-directory anti-patterns have been resolved across admin/, context/, hooks/, utils/, settingsModal/sections/, plc/ tree, and components/common/library/. Future D4 runs should focus on staleness checks for newly added files only.

### D5 â€” Toast Architecture in Admin Components

**Scope:** Admin panels, configuration modals, and other components that surface in-app notifications.

**Canon:** `addToast(message, type)` via `useDashboard()` â€” the centralized toast queue rendered by DashboardView's ToastContainer.

**Acceptable exception (local Toast component):** Several admin modal components use `<Toast message="..." type="..." />` from `components/common/Toast` with local `useState`. This pattern is acceptable when:

1. The component may render outside a DashboardProvider context (e.g., student-facing or subs routes), OR
2. The notification is **inline within the modal body** â€” the `<Toast>` renders inside the modal's JSX tree, adjacent to content rather than as a global overlay. This is intentional UX because stacked z-index layering means a global `addToast` would appear behind the modal overlay.

**Do not unify the local Toast pattern blindly** â€” confirm whether `useDashboard()` is available in the component's render tree before replacing with `addToast`. If it is available AND the toast is used as a global overlay (not inline in modal body), prefer `addToast`.

**Status (run 4):** D5 is exhaustively documented. Two new admin components found during run 4 staleness scan â€” `SaveAsTemplateModal.tsx` (D5-E15) and `Organization/OrganizationPanel.tsx` (D5-E16) â€” both confirmed intentional exceptions (inline modal body and full-panel manager respectively). No further admin component conversions are expected.

## Notes & Gotchas

- **`ScaledEmptyState` conversions inside a nested sub-panel need a container-scope check, not just a visual eyeball (run 26):** `ScaledEmptyState`'s `cqmin` sizing resolves against the nearest ancestor with `container-type: size` â€” which today is set exactly once, at the whole-widget level (`WidgetRenderer.tsx`/`DraggableWindow.tsx`), not re-established at any nested sub-panel. Converting a hand-rolled empty state that lives inside a smaller bordered sub-panel (e.g. `ActivityWall/Widget.tsx`'s submissions card) makes the icon/text scale off the _full widget_, not the smaller panel â€” a real, visible size jump (in this case the icon roughly doubled, 24pxâ†’48px cap at default widget dimensions), not just a subtle rounding difference. `pnpm run validate`/`build` cannot catch this â€” it's a runtime CSS-scoping issue, not a type or test failure. Before converting any empty state that sits inside a sub-panel narrower than the full widget (as opposed to being the widget's only/full-bleed content), check whether that sub-panel establishes its own `container-type: size`; if not, treat the conversion as VISUAL-RISK requiring either a properly-scoped nested container boundary (careful â€” rescoping affects every other `cqmin` user in that panel, e.g. populated-state content) or a size-override affordance on `ScaledEmptyState`, not a straight swap. `ActivityWall/Widget.tsx:1877` hit this in run 26 (PR #2143, closed without merging after review caught it).
- **Background subagents (async `Agent` calls) can get stuck in a "wait for notification" loop when they themselves spawn a background process (run 26):** 3 of 5 dimension subagents this run ended their turn saying some version of "I'll pause here and wait for the validate command / monitor to notify me" â€” but nested background-process completion inside a subagent's own context doesn't generate a wake-up the way the top-level orchestrator's task-notifications do, so each just sat inert until resumed via `SendMessage`. Fix: when resuming a stuck agent, explicitly instruct it to block synchronously (`wait <PID>` or a polling loop, or just re-run the command in the foreground) and not end its turn until it has the actual result in hand. Consider adding a line to future dispatch prompts: "run validate/build in the foreground, not backgrounded â€” don't end your turn waiting for a notification that won't arrive."
- **Worktree staleness recurred a 2nd time, same night as a clean run (run 25):** The D3 subagent's worktree was again cut from a stale base (`0adda31`, ~30 commits behind `origin/dev-paul`'s `9e08914` â€” the same commit seen stale in run 24), while the D4 subagent's worktree that same run was correctly based on current `dev-paul`. This confirms the staleness is per-worktree-creation, not a systemic clock/cache issue â€” always check each worktree independently via `git merge-base HEAD origin/dev-paul` before trusting its branch, even when a sibling worktree from the same dispatch was fine. Repair procedure (same as run 24): extract the diff, discard the stale worktree/branch, cut a fresh branch from current `dev-paul`, reapply, re-validate.

Skip it when the label doesn't map to one clear control (e.g. "Available To" above a description + button grid, or "PDF File" above a button that triggers a hidden file input) â€” `htmlFor` on an ambiguous target is worse than omitting it.
